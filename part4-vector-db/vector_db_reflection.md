## Vector DB Use Case

A traditional keyword-based database search would help to some extent, but it would not be sufficient on its own for a system where lawyers ask questions in plain English across 500-page contracts. Keyword search mainly looks for exact words or close matches. That creates a problem because legal language is often complex, highly varied, and full of synonyms. For example, a lawyer may ask about "termination clauses," while the contract may use phrases like "right to end this agreement," "grounds for cancellation," or "expiration and exit terms." A keyword system may miss these relevant passages if the wording does not match closely.
A vector database improves this by enabling semantic search. Instead of focusing only on exact words, it stores embeddings, which are numerical representations of meaning. When a lawyer asks a question in plain English, that question is also converted into an embedding. The vector database then finds the contract passages that are closest in meaning, even if the wording is different. This makes the search much more intelligent and practical for large legal documents.
In such a system, the contract would usually be broken into smaller chunks, and each chunk would be embedded and stored in the vector database. When a query is asked, the most relevant chunks are retrieved and shown directly or passed to an AI model for answer generation. So, the best approach is not keyword search alone, but a vector database-based semantic retrieval system, possibly combined with keyword search for precision.
